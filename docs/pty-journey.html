0. 起点：为什么做、初版错在哪

CCM（Claude Code Manager）原本每个回合执行一次 claude -p "<prompt>" --output-format stream-json --resume <sid>。 简单可靠（进程退出=回合结束、exit code=成败），但每条 chat 消息、每个 loop 迭代都要整进程冷启动 + 从 JSONL 重放全部历史，会话越长越慢，且无法中途注入消息或软中断。

目标：一个任务一个常驻交互式 CC 进程，多轮对话同进程完成。设计参考了 Teleos（zjw49246/Agent2Agent）的生产验证经验——深度调研它后得到本框架的核心哲学：

PTY 只当宿主，消息走协议层。 伪终端只负责进程保活、启动应答、Esc 中断；输入走 MCP channel 注入、输出读 CC 自己写的 JSONL。 凡是试图「模拟人类打字、解析终端 ANSI 输出」的路都是死路。

初版框架（重构前）恰恰踩了这些死路，带着 4 个结构性缺陷（见第 3 节）， 而且 channels 注入链路整体是坏的却没有任何测试发现——这是后来一切问题的源头。

1. 初版搭建与第一批问题（task 77，05:45–15:58，约 10 小时）

一切从一份文档开始：把 CCM 仓库的 docs/pty-interactive-mode.md 喂给 Claude 深度讲解，随即落地为独立项目——初始化 /home/ubuntu/Projects/PTY（CLAUDE.md、GitHub remote、 文档入库），并系统调研了 PTY 的固有坑（kernel buffer ~4KB、pty.openpty vs pexpect、轮询延迟等）。期间另做了一轮开源生态扫描（612 行报告 docs/pty-open-source-research.md），确认 PTY+JSONL 是所有认真项目的共同路线， 其中 Teleos (zjw49246/Agent2Agent) 的「PTY 只当宿主、消息走 MCP channel」成为最重要的架构参考。

初版框架跑起来后，真实任务（task 45：给会话发文件）立刻撞上第一个大坑：

问题 0resume 3 秒超时崩溃——PTY 模式的第一个生产事故
现象
task 45 发消息后卡死，PTY session 崩溃：Session 06c16dac process died (pid=547800), alive=False。JSONL 显示 CC 在 tool 执行中途消失，随后出现 last-prompt / ai-title / mode 等新 session 初始化元数据——进程死了又被重启
根因
CC 的 --resume 有 3 秒 stdin 超时：恢复一个处于"等待输入"状态的 session 后，3 秒内收不到输入就直接退出。而初版实现是两步走——先 spawn(resume_id)、过一会儿再 send_prompt()，间隔常超 3 秒
解法
initial_prompt 贯通三层（commit cecf81c）：Session.start(initial_prompt=...) 在 spawn 的同一时刻投递 prompt，pool.get_or_create 与适配层 launch() 一路传下来。配合 _auto_resume 兜底，task 45 复测成功跑完（15:58）

task 77 同时识别出三个待解的体验问题，成为后续 Phase 的待办清单： thinking 加密不可见（第 7 节解决）、JSONL 轮询延迟（300ms 轮询权衡）、 resume 时序脆弱（Phase 1 的 channel 注入主路从根上消除）。

定位结论：claude-pty 是唯一同时具备 「PTY 宿主 + Channel 注入 + 会话池 + 流式事件 + CCM 权限/号池整合」的 Python 方案。

2. Phase 0：动手前的四个验证（半天）

重构前先用真实 CC 做了 4 个 spike，每个都直接改变了设计，4/4 通过：

#验证结论与影响
1channel 注入能否唤起新 turn（承重墙） ✅ idle 会话收到 MCP notification 自动开新 turn。但发现 dev-channels 确认对话框会卡死启动——这正是旧版 channels 全坏的根因
2交互模式 JSONL 事件形态 没有 result 事件（-p 专属）；意外发现 system/turn_duration 哨兵：每 turn 恰一条、在所有消息之后——比 end_turn 可靠得多

4预写 trust 配置 ✅ spawn 前往 .claude.json 写 hasTrustDialogAccepted 等字段，对话框从根上不出现
方法论教训：对 CC 内部格式的任何假设必须先用真实进程 dump 验证。 逆向第三方路径/格式规则时要用边界字符做实验（. _ 空格 @ 大写），不要只验证 happy path。

3. Phase 1：核心重构——修掉 4 个结构性缺陷

对应 commits 1f889cc（I/O 核心修复）、0c64014（输入通路切换）、 3b2546c（文档），+495/-58，单测 92/92 ✅，集成测试 6/6 ✅。

问题 1PTY stdin 逐字符输入（危险且极慢）
现象
初版逐字符写 stdin + 每字符 sleep 模拟人类打字：UTF-8 多字节字符被按"字符"拆分发送（中文/emoji 场景脆弱）；prompt 含 \n 被 TUI 当作提交键提前发送；1 万字符要 8 分钟
根因
把 PTY 当成了消息通道，直接对抗 TUI 输入层的所有复杂性（快捷键、斜杠补全、粘贴处理）
解法
输入主路改 channel 注入（MCP notification，完全绕开 TUI，失败重试 5 次）；stdin 仅作 fallback 且改为 bracketed-paste 整段写入（\x1b[200~ ... \x1b[201~），换行不再触发提交
问题 2回合结束检测用 stop_reason=end_turn（提前截断）
现象
同一 turn 的 thinking 块和 text 块各占一条 JSONL 行、都带 end_turn，首条命中即截断事件流，尾部消息丢失
解法
改用 spike 发现的 {"type":"system","subtype":"turn_duration"} 哨兵——交互模式每 turn 恰好一条且在所有消息之后。三层兜底：哨兵 → response_timeout（30 分钟）→ dispatcher 超时 kill
问题 3JSONL 路径推导规则错误（输出通路全盲）
现象
旧实现 cwd.replace("/","-").replace("_","-") 漏掉 ./空格等字符，cwd 含这些字符时轮询的文件根本不存在，永远读不到事件
解法
按实测改为 re.sub(r"[^A-Za-z0-9]", "-", cwd)
问题 4启动对话框枚举触发词（channels 整体瘫痪的根因）
现象
--dangerously-load-development-channels 启动会弹「I am using this for local development / Enter to confirm」确认框。旧触发词列表（trust/safety/…）不含它 → CC 永远卡在对话框 → channel MCP server 从未启动 → inject 全部 Connection refused，整条注入链路自始至终是坏的，且无测试发现
根因
枚举具体对话框文案必然漏；且 CC 用光标定位渲染，可见的"空格"并非真空格，朴素子串匹配也匹不上
解法
双保险：(a) spawn 前预写 .claude.json 的 projects[cwd]（hasTrustDialogAccepted + enabledMcpjsonServers），从根上消灭对话框；(b) drain loop 改为通用结构匹配——剥 ANSI + 折叠空白后找 Entertoconfirm（所有确认框的统一尾缀），600ms 冷却后主动 re-check（连续第二个对话框渲染完后 CC 静默，没有新输出驱动检查）。并补上端到端集成测试 TestChannelInjectionIntegration
教训：任何"理论上能用"的路径必须有端到端集成测试。 channels 链路坏了一整个版本周期，单元测试全绿。

4. Phase 2：接入 CCM——feature flag + 单一分流点

CCM 侧（PR #24）核心设计：零改动原 -p 路径，单一分流点 + 数据库开关，可随时安全降级。

# instance_manager.launch() 单一分流点
if provider == "claude" and use_pty_mode:
    return CCMBackend.launch_for_ccm(...)   # PTY 常驻会话
else:
    return subprocess("claude", "-p", ..., "--output-format", "stream-json")  # 原样
  • 后端：GlobalSettings 表新增 use_pty_mode 列（migration）+ GET/PUT /api/settings/runtime
  • 前端：导航栏滑动开关（时区选择器左侧）；语义为只影响新启动的任务，关闭时执行中的 PTY 会话跑完为止，idle 会话由 drain_idle() 立即回收（commit 0b59e90）
  • 适配层（commit a02289e / a478051）：CCMBackend 自持 BridgeHub、池内会话默认开 channels、resume_existing 支持冷恢复、proxy kill 联动会话拆除、真实 PTY pid 透出给 Instance.pid
  • 事件对齐：PTYEvent 与 CCM StreamParser 期望格式 1:1，下游（dispatcher / log_entries / WebSocket）零改动

验证：CCM 单测 66/66 ✅，tsc + vite build ✅，开关 GET/PUT 往返与重启恢复实机验证 ✅。

5. 生产实测暴露的问题群（4 个真 bug + 2 个观测盲区）

接入后用真实任务（task 47/48 等）实测，连环暴露 4 个真 bug， 集中在 commit 30b6588 + CCM 97565d1 修复：

问题 5回复黑洞——用户只看到「…我把解释放在这里」
现象
chat 回复内容极少甚至没有，正文神秘消失
根因
channel 注入附带了 pty_bridge_reply 工具 + 指示语，CC 误以为要"用工具回信"，把真实回答发进了无人消费的通道；JSONL 里只剩一句过场白
解法
删除 reply 工具；指示语改为「channel 消息 = 用户消息，直接在对话中回答」。回答自然落回 JSONL，被轮询器照常采集
问题 6冷恢复卡死——30 分钟超时占死队列
现象
resume 已有 session 的任务全部卡死直到超时
根因
spawn 的瞬间就往 stdin 写 prompt，此时 TUI 尚未就绪，输入被吞 → turn 永远不开始 → 哨兵永远不来
解法
删除 spawn 时的 stdin 投递，所有首条消息统一走 channel 注入（注入本身带重试，天然等到 MCP server 就绪）
问题 7会话池尸体——中断后下一次 launch 撞死会话
现象
mid-turn 手动停止任务后，后续任务复用同 session 失败
根因
kill 掉进程但池里仍留着死引用，get_or_create 返回尸体
解法
stop / kill / 超时三条路径全部联动池清理；mid-turn 永不驱逐改为「死亡即除名」
问题 8orphan CC 进程——服务重启后旧进程还占着 session
现象
CCM 重启后，前世的 CC 进程仍存活并持有 session 文件
解法
CCM lifespan shutdown hook 调 pty_backend.shutdown()，关停时统一回收全部 PTY 会话（CCM 97565d1）

顺手修掉的两个观测盲区：

  • 日志黑洞：claude_pty 的 logger 没配 root handler，所有日志静默丢弃 → 加 basicConfig，问题 5–8 的定位全靠补上的日志
  • 额度显示 "unknown"：交互模式 JSONL 无 contextWindow 字段 → 按 model 名回填默认值，用量改从 usage 累加

6. 撞限检测与号池换号（Phase 3 重头戏）

这是 -p → PTY 迁移中最隐蔽的能力缺口：-p 模式撞限时进程退出、exit code 非零， CCM 的号池轮转（检测 → 冷却 → 换号 → 重拉）全靠这个信号； 而交互模式撞限时进程根本不退出，只在 TUI 上显示一条横幅，PTY turn 会一直挂到超时。

问题 9撞限文案正则过时（CCM PR #26）
现象
新版 CC 的限流文案（如「You've hit your session limit · resets X:XXpm」）不匹配旧正则，-p 模式也换不了号
解法
更新正则，5 条真实文案做回归测试；修复默认账号无法参与轮转的 dispatcher 分支。142 个单测通过
问题 10PTY 模式撞限无退出信号（PTY 6d788f7）
现象
交互式 CC 撞限不退出，PTY turn 挂死到 30 分钟超时，无法触发换号
解法
四层联动： ① pty_process 常开折叠空白扫描，识别限流横幅（hit your session/weekly limit、usage limit reached）→ rate_limited 标志； ② jsonl_reader 归一化结构化 rate_limit_event 行为 error 事件； ③ session 提前结束 turn，错误消息内嵌「usage limit reached」喂给 CCM 既有检测正则； ④ 适配层把撞限 turn 翻译成 exit_code=1 触发宿主轮转； ⑤ 池在 config_dir 变化（换号）时强制重建会话，绝不热复用撞限账号的会话。 CCM 侧零改动——既有轮转链路原样工作

7. 收官：loop 热复用与 thinking 可见性

7.1 Loop 热会话复用（CCM PR #27）

PTY 模式的本意就是免冷启动，但 loop 最初每轮迭代都新建会话，等于白做：

# 之前（每轮新建，上下文无法累积）
迭代 0: 新建会话 → turn 1 → 释放 ❌
迭代 1: 新建会话 → turn 1 → 释放 ❌

# 之后（一个会话、一轮一个 turn）
迭代 0: 新建常驻会话 → turn 1 → 信号(continue)
迭代 N: 同一会话注入 → turn N+1 → 信号(done)
结束:   会话释放 ✅   上下文跨迭代连续、免冷启动

实测收益：第一轮（冷启动）约 25s，第二轮（热复用）7.8s。 77 个相关测试通过，含「-p 模式行为零变化」的专用断言。

7.2 Thinking 可见性（e4ac6ee）

交互模式 JSONL 里 thinking 一度只有加密 payload，UI 无法显示。 后确认 CC 配置 showThinkingSummaries 可让 JSONL 带明文 thinking——问题解决，无需任何解析 hack。

9.2 输入通路：channel 注入为主，stdin 为备

send_prompt(text)
  ├─ 主路：BridgeHub.inject(sid, text)            # 重试 5 次
  │     → HTTP POST /inject
  │     → channel_server（MCP stdio 子进程）
  │     → notifications/claude/channel 推给 CC
  │     → CC 收到后自动开新 turn（Phase 0 spike 验证）
  └─ 备路：PTY stdin（仅注入彻底失败时）
        → bracketed-paste 包裹整段写入（\x1b[200~ … \x1b[201~）

channel 由 spawn 参数自动启用（--dangerously-load-development-channels server:pty-bridge + .mcp.json 注册 claude_pty.channel_server），其确认对话框被预写 trust + 通用应答双保险消灭。

9.3 输出通路：JSONL 轮询 + 哨兵收尾

  • 路径：~/.claude/projects/<re.sub(r'[^A-Za-z0-9]','-',cwd)>/<session_id>.jsonl
  • JsonlReader 行缓冲处理 partial write，normalize 跳过 6 种噪声事件，产出与 CCM StreamParser 对齐的事件
  • 回合结束 = system/turn_duration 哨兵（交互模式每 turn 恰一条、在所有消息之后；没有 result 事件，cost 从 usage 累加）
  • 三层兜底：哨兵 → response_timeout → dispatcher 超时 kill

9.4 PTY 的真正职责（只有四件事）

职责机制
进程保活pty.openpty() + Popen，常驻交互式 CC
启动应答drain loop 剥 ANSI + 折叠空白匹配 Entertoconfirm，600ms 冷却 re-check
软中断写 Esc 字节，CC 当前 turn 优雅终止
带外信号_last_output 活动信号；限流横幅扫描 → rate_limited

9.5 会话池与生命周期语义

# 关闭 PTY 开关的完整语义
set_pty_mode(False)
  ├─ 新任务立即走 -p
  ├─ 执行中的 turn 跑完为止（不中途杀）
  ├─ idle 会话立即回收（drain_idle）
  └─ 死亡/中断/超时全路径联动池清理（无尸体）

# 撞限换号
检测（横幅/JSONL/正则三路）→ turn 以 exit_code=1 结束
  → CCM 既有轮转：冷却 → 选新账号（新 config_dir）
  → 池发现 config_dir 变化 → 强制重建会话 → --resume 续上下文
