PTY 框架

问题定义 方法介绍 意义
一个独立的框架：在 Claude Code 交互模式外层套一个伪终端（PTY）， 让原本只能由人在终端里使用的交互模式，变成可被程序驱动的自动化运行时。

1问题定义：我们遇到了什么问题

6 月 15 日之后，Claude Code 的 -p（非交互）模式额度大幅下降。 我们目前所有的项目都构建在 -p 模式之上——额度一降，所有项目都无法运转。

这不是某一个项目的优化问题，而是全线项目的生存问题。我们急需一套框架来解决它。

出路只有一条：额度政策对交互模式友好，所以必须改用 Claude Code 的交互模式。 但交互模式是为「人坐在终端前」设计的，对程序化调用极不友好：

-p 模式给程序的东西交互模式的现实
stdin 传 prompt，一次性输入要经过 TUI（快捷键、补全、粘贴处理、确认对话框）
stdout 输出结构化 stream-json屏幕输出是 ANSI 转义序列渲染的"画面"，不可解析
进程退出 = 回合结束，exit code = 成败进程常驻不退，没有任何显式的"回合结束"信号
额度撞限 → 进程退出报错撞限只在屏幕上画一条横幅，进程照常活着
不需要终端必须有一个真实的 TTY 才肯启动

而且交互式 CC 拒绝在普通管道上运行——它要求一个真实终端。 所以方案的形态被问题本身决定了：在交互模式外层套一个伪终端（PTY）， 由框架扮演"坐在终端前的人"，把交互模式重新包装回程序可用的接口。 这就是 PTY 框架。

2方法介绍：我们是怎样解决这个问题的

2.1 核心思路：PTY 只当宿主，消息走协议层

最直觉的做法是"全仿真"：往 PTY 里逐字符打字模拟输入，解析 ANSI 屏幕输出拿结果。 我们验证过，这是死路——UTF-8 多字节字符会被拆坏、换行被当成提交键、一万字符要打 8 分钟， 而屏幕输出根本无法可靠还原成结构化数据。

我们的设计：PTY 只负责"让进程活着"，真正的输入输出都绕开终端，走协议层。
输入——通过 MCP channel 把消息以 notification 形式直接注入 CC 的上下文，唤起新一轮对话；
输出——不看屏幕，直接读 CC 自己实时写盘的 session JSONL，拿到的天然就是结构化事件。
调用方任何需要驱动 Claude Code 的程序
↓ send_prompt() / stream_events() / interrupt()
Session / SessionPool常驻会话封装 · LRU 池 · 免冷启动复用
↓
PTY 宿主进程保活 · 自动按掉启动对话框
Esc 软中断 · 撞限横幅扫描
输入：channel 注入MCP notification 直入上下文
绕开 TUI，自动开新 turn
输出：JSONL 轮询读 CC 实时写盘的事件流
300ms 增量 · 结构化
↓ 结构化事件流（与 -p 模式的 stream-json 对齐）
调用方原有的下游逻辑，零改动

PTY 层只做四件事：进程保活、启动应答（预写 trust 配置 + 通用对话框自动确认双保险）、 Esc 软中断、带外信号（活动时间戳与撞限横幅扫描）。 回合边界用 JSONL 里的 system/turn_duration 哨兵判定——实测它每个 turn 恰好出现一条、 且在所有消息之后，是交互模式下唯一确定性的"回合结束"信号。 这些关键假设（注入能唤起新 turn、哨兵的存在、JSONL 路径规则）全部先用真实 CC 进程做 spike 验证后才施工。

2.2 解决的三个关键问题

①Thinking 内容的获取

本来通过终端方式是无法获得 thinking 内容的——交互模式的 JSONL 里 thinking 只有加密 payload， 屏幕上更是看完即逝。这曾被我们列为"无解"的限制。

后来我们发现：在 settings 文件里配置 showThinkingSummaries 之后， JSONL 里直接带明文 thinking，框架照常采集，不需要任何解析 hack。

这对我们十分重要：模型的思考过程是轨迹中最有价值的部分， 拿到完整 thinking 意味着我们采集的每一条轨迹都包含「思考 → 行动 → 结果」的全链条。

②对运行速度的影响：不降反升

-p 模式每个回合都要付"冷启动税"：整进程启动 + 从历史文件重放全部上下文，会话越长越慢。 PTY 框架是常驻进程，上下文一直热着，新消息注入即开新 turn。

~25s-p 模式一个回合（冷启动 + 重放历史）
7.8sPTY 模式同等回合（热会话实测）
3×+多轮场景提速，会话越长差距越大

也就是说，这次迁移不是"为了活下去而忍受降级"——速度反而比原来更快。

③任务状态的流转：几乎无感

上层系统对一个任务的状态机依赖三类信号：回合何时结束、任务成败、异常（撞限/超时/中断）。 这些信号在 -p 模式下全部来自"进程退出"，而交互模式进程不退——框架为每个信号都构造了等价物：

状态信号-p 模式的来源PTY 框架的等价物
回合结束进程退出 + result 事件turn_duration 哨兵（+ 超时双兜底）
成败/退出码exit code按 turn 结果合成 exit code
额度撞限进程退出 + stderr 文案PTY 输出横幅扫描 + JSONL 信号 → 合成报错与非零退出码，上层换号轮转逻辑原样工作
中断杀进程（会话作废）Esc 软中断——只打断当前 turn，会话保活，下一条消息接着用
用量/成本result 事件直接给逐 turn 累加 usage 字段

结论：使用 PTY 框架对任务状态的流转不会产生很大的影响——上层看到的事件流和信号 与 -p 模式对齐，状态机不用重写；中断语义甚至变好了（软中断不再报废整个会话）。

2.3 额外解决的问题：-p 模式结构上做不到的能力

以下能力不是迁移的目标，而是常驻会话架构的自然产物——但每一项都是 -p 模式结构上不可能支持的：

能力-p 模式PTY 框架
执行中注入消息启动后即黑盒，只能等整个回合跑完channel 注入支持任务执行中途插入新指令——跑偏了立即纠偏，不必等它把错误路线走完
软中断中断 = 杀进程 = 会话作废重来Esc 只打断当前 turn，上下文完好，下一条消息接着用
多轮上下文累积循环任务每轮失忆重启，靠 prompt 塞摘要同一会话一轮一个 turn，第 N 轮天然记得前 N−1 轮
额度利用效率每回合冷启动，prompt cache 全 miss常驻会话连续 turn 落在 cache TTL 内，命中率远高于冷启动——同样的额度跑更多任务（定量对比待做，方向确定）

其中第四点在"额度大幅下降"的背景下是第二层生存价值：不仅能用交互模式，还把每一分额度用得更省。

2.4 工程可靠性：踩过的坑都已闭环

从初版到生产可用，我们定位并修复了一系列问题，每一个都有复现、根因和回归测试：

问题根因 → 修复
resume 3 秒超时崩溃CC 恢复会话后 3 秒收不到输入即退出 → spawn 同刻投递 initial_prompt
启动对话框卡死枚举文案必然漏 → 预写 trust 配置 + 剥 ANSI 折叠空白后按结构特征（"Enter to confirm"）通用匹配
回合提前截断误用 end_turn 判回合 → 改 turn_duration 哨兵
输出全盲JSONL 路径推导规则错 → 实测逆向出真实规则（所有非字母数字 → -）
回复进了"黑洞"注入附带的回信工具误导模型 → 删工具，注入消息即用户消息
会话池残留死会话kill 后池里留尸体 → 死亡即除名，全路径联动清理
撞限挂死交互模式撞限不退出 → 横幅扫描合成退出信号（见 2.2-③）

全部修复均有单元 + 端到端测试覆盖（累计 400+ 用例），并在多账号、长会话、循环任务的真实负载下验证过。

3这个东西对我们的意义

所有项目不需要大改，就可以无痛地从 -p 模式迁移到交互模式，并且拿到比以前更丰富的轨迹。

意义说明
生存6 月 15 日之后 -p 模式额度不再可用，而我们的全部项目可以原样跑在交互模式上——框架对上层暴露的事件流和信号与 -p 对齐，迁移成本接近零
更好的运行时不是降级求生：常驻会话免冷启动（7.8s vs 25s）、可中途注入消息、可软中断不报废会话——这些都是 -p 模式做不到的
更丰富的轨迹完整 thinking + 工具调用 + 结果的全链条结构化轨迹，是后续一切数据工作的原料
通用的方法论资产「PTY 当宿主 + 消息走协议层 + 读结构化日志」不绑定 Claude Code——任何交互式 CLI agent（Codex CLI、Gemini CLI…）都适用同一套打法。哪家的非交互模式收紧，都可以再套一次，这是可复用的框架而非一次性补丁
逆向 know-how 壁垒JSONL 路径规则、turn_duration 哨兵、resume 3 秒超时、trust 预写、thinking 配置——这些没有任何官方文档，全靠真实进程实验逆向而得。"少数能做到的团队"，护城河正是这堆细节

交互模式的程序化驱动有相当的技术门槛——真实 TTY、TUI 对话框、回合边界、撞限信号， 每一关都能挡住一批团队。在 6 月 15 号之后，我们可能会成为少数能够做到这一点的团队。
