1. 一句话理解：-p 模式 vs PTY 模式

-p 模式（现在）

每次发 prompt 都新启动一个 CC 进程，等它处理完，再杀掉。多轮对话靠 --resume 参数恢复上下文。

就像每次想聊天都要重新打开 APP，打开要等 5-8 秒，聊完就关掉。

PTY 模式（本框架）

CC 进程一直开着，你想发 prompt 就直接打字进去，它立刻响应。多轮对话天然连续。

就像一直开着聊天窗口，想说什么就说什么，对方立刻回复。

对比维度-p 模式PTY 模式
每次 prompt 开销5-8 秒冷启动即时响应
多轮上下文需要 --resume + session 文件天然连续，进程一直活着
中途注入消息不可能Channels MCP 支持
并发 session每个 prompt 独立进程LRU 会话池复用进程
输出读取解析 stdout 的 stream-json读取官方 JSONL session 文件

2. 什么是 PTY？为什么需要它？

当你在终端里运行 claude 命令，CC 启动了一个交互式 TUI（终端界面）——有光标、有输入框、会检测你的按键。 它怎么知道自己在一个"终端"里？因为它检查了自己的 stdin/stdout 是不是连着一个 TTY（终端设备）。

问题来了：如果你的 Python 程序用 subprocess.Popen 启动 CC，stdin/stdout 连的是 管道（pipe），不是终端。CC 检测到这一点，会拒绝进入交互模式，因为它的 TUI 没法在管道上画界面。

PTY（Pseudo Terminal，伪终端） 就是解决方案。它是操作系统提供的一对设备：
• master 端：你的 Python 程序拿着，往里写字符 = "模拟键盘输入"，从里读数据 = "读屏幕输出"
• slave 端：CC 拿着，它看到的是一个真正的终端设备

CC 完全不知道对面不是人类在打字。它开开心心进入交互模式，画它的 TUI，等输入。
你的 Python 程序 Claude Code 进程 往 master_fd 写字符 ─────→ slave 端（看起来是真终端） "实现登录功能\r" CC 觉得："有人在键盘上打字" CC 开始处理 prompt CC 把响应写到 stdout 从 master_fd 读数据 ←───── slave 端输出（ANSI 转义 + TUI 渲染） （但我们不解析这个输出，太乱了） 同时，CC 把结构化数据写到 JSONL 文件 我们直接读 JSONL 文件获取干净的响应 ←── 这才是我们真正的数据源

3. Step by Step：一次 prompt 是怎么跑完的

用户代码调用了 session.send_prompt("实现登录功能")，背后发生了什么：

阶段一：进程启动（只在第一次发生）

  1. 1
    创建 PTY 对 调用 pty.openpty() 拿到 master_fd 和 slave_fd。设置终端大小为 50 行 × 200 列。
  2. 2
    清洗环境变量 删除所有含 CLAUDE、CLAUDECODE、AI_AGENT 的环境变量（防止干扰），设置 TERM=xterm-256color 让 CC 认为在一个正常终端里。
  3. 3
    启动 CC 进程 用 subprocess.Popen 启动：claude --dangerously-skip-permissions --session-id {uuid}，stdin/stdout/stderr 都连到 slave 端。CC 看到自己在一个终端里，进入交互模式，显示 TUI 欢迎界面。
  4. 4
    启动 drain 线程 一个后台守护线程，每 50ms 从 master_fd 读一次数据，把 CC 的 TUI 输出"排掉"。这一步至关重要——原因在后面解释。
  5. 5
    等待 CC 就绪 等 12 秒（可配置），让 CC 完成初始化。这段时间 CC 在加载模型配置、MCP 服务器等。初始化产生的 JSONL 消息被跳过。

阶段二：发送 prompt

  1. 6
    逐字符写入 prompt 到 master_fd 把 "实现登录功能" 一个字一个字地写进 PTY。每个字符之间等 30-70 毫秒的随机间隔。

    为什么不能一次性写入整个字符串？因为 CC 的 TUI 用了 raw mode——它在监听每一个按键事件。 如果你一次性把整个字符串灌进去，CC 的输入缓冲区会乱掉，第一轮可能没问题，第二轮就会吞字符。 逐字符加随机间隔，等于在模拟一个人在打字。
  2. 7
    写完后发送回车 \r 告诉 CC "输入完了，可以处理了"。CC 开始生成响应。

阶段三：读取响应

  1. 8
    每 300ms 轮询 JSONL 文件 CC 在处理请求时，会把每一步的结构化数据写入 JSONL session 文件 （路径：~/.claude/projects/{项目路径hash}/{session_id}.jsonl）。

    我们不去解析 PTY 的输出（那里面全是 ANSI 颜色码、光标移动、进度条动画等等，根本没法用）， 而是直接读这个 JSONL 文件。每行是一个 JSON 对象，包含消息类型、内容、工具调用等完整信息。
  2. 9
    把每条 JSONL 消息转换成标准事件 yield 出去 JSONL 文件里的格式长这样：{"type":"assistant","message":{"content":[...]}}

    我们的 normalize() 函数把它转换成统一的 PTYEvent，输出格式跟 CCM 的 StreamParser 完全一致。 这样 CCM 现有的事件处理代码 一行不用改 就能直接对接。
  3. 10
    检测到 stop_reason == "end_turn"，响应结束 CC 写了一条 type=assistant 且 stop_reason=end_turn 的 JSONL 行，说明它说完了。 等 3 秒让 CC 回到输入状态，准备接收下一条 prompt。
时间线 0s 你的代码: session.send_prompt("实现登录功能") ├── PTYProcess: 逐字符写入 "实现登录功能" + "\r" │ 每个字符间隔 ~50ms，总共 ~0.3 秒 0.3s │ │ CC 开始处理... │ 0.6s JsonlReader: 第 1 次轮询 JSONL，发现新的 assistant 消息 ├── yield PTYEvent(type=thinking, content="让我思考...") │ 1.2s JsonlReader: 第 2 次轮询 ├── yield PTYEvent(type=message, content="好的，我来实现...") │ 3.0s JsonlReader: 第 N 次轮询 ├── yield PTYEvent(type=tool_use, tool_name="Edit") │ 5.0s JsonlReader: 发现 tool_result ├── yield PTYEvent(type=tool_result, content="文件已修改") │ 8.0s JsonlReader: 发现 stop_reason == "end_turn" ├── yield PTYEvent(type=result, cost_usd=0.04) │ 8.0s send_prompt 迭代结束 等 3 秒让 CC 回到输入状态... │ 11.0s 可以发下一条 prompt 了

你的代码看到的

    print(event.event_type, event.content)

# tool_use   Edit /src/auth.py
# tool_result 文件已修改
# message    "登录功能已实现，包括..."
# result     cost_usd=0.04

4. 四个会踩的坑和我们怎么解决的

坑 1：PTY 缓冲区只有 4KB，不排空就死锁

问题： Linux PTY 的内核缓冲区只有 4KB（N_TTY_BUF_SIZE）。CC 输出的 TUI 渲染数据非常多（ANSI 转义码、进度条、代码高亮……），很快就能把 4KB 填满。一旦缓冲区满了，CC 的 write() 系统调用就会 阻塞——CC 整个进程挂住，不再处理任何事情。
解决： 启动一个独立的 daemon 线程，每 50ms 执行 select() + os.read(65536)，把 CC 输出的数据全部读走扔掉。 我们不需要这些数据（TUI 渲染对我们没用），但必须一直"排水"，不让缓冲区满。

这个线程是纯同步的，不碰 asyncio，所以不会有事件循环冲突。它就像一个 "7×24 小时开着的排水泵"。
如果没有 drain 线程： CC 进程: write("ANSI渲染数据...") → write() → write() → 缓冲区满！write() 阻塞！ CC 挂住了，什么也不做 你的 prompt 永远不会被处理 有 drain 线程： CC 进程: write("ANSI渲染数据...") → write() → write() → 正常写入，继续工作 │ │ │ drain: read() 读走 read() 读走 read() 读走 数据直接丢弃 数据直接丢弃 数据直接丢弃

坑 2：逐字符输入，不能批量写入

问题： CC 的前端是 Ink（React for CLI），运行在 terminal raw mode。在 raw mode 下，终端逐个按键地传递给应用，没有行缓冲。如果你一次性把 "实现登录功能\r" 整个字符串写入 PTY，CC 的输入处理在第一轮可能凑合能用，但第二轮开始就会出现吞字符、缓冲区错乱的问题。
解决： 一个字符一个字符地写，每个字符之间等一个高斯随机分布的延迟（均值 50ms，标准差 20ms，钳位在 10-150ms）。 相当于模拟一个打字速度中等的人类。写完最后一个字符后额外等 100ms，再发回车 \r。 
# 实际的 send_prompt 核心逻辑（简化版）
for ch in "实现登录功能":
    os.write(master_fd, ch.encode("utf-8"))
    time.sleep(random.gauss(0.05, 0.02))  # 30-70ms 随机间隔
time.sleep(0.1)
os.write(master_fd, b"\r")  # 回车：提交输入

坑 3：JSONL 文件 partial write，读到半截数据

问题： CC 往 JSONL 文件写数据时，一行可能分多次 write() 系统调用完成。如果我们读文件的时机刚好夹在两次 write 之间，就会读到半截 JSON，比如 {"type":"assis —— 直接 json.loads() 就炸了。

更糟的是，如果你把这个半截数据丢弃，下次读到后半段 tant","message":{...}} 也拼不起来了。消息永久丢失。
解决：行缓冲机制
读到的数据按 \n 分割。最后一段如果没有以 \n 结尾，就是不完整的行，存到 _buffer 里。 下次读取时，新数据拼接在 buffer 后面再分割。只有完整的行（以 \n 结尾的）才会被 JSON 解析。
第一次读取： 文件内容: {"type":"assistant",...}\n{"type":"us ↑ 这里被截断了 完整行: {"type":"assistant",...} → 解析，返回 buffer: {"type":"us → 保留，等下次 第二次读取： 新数据: er","message":{...}}\n 拼接: {"type":"user","message":{...}}\n 完整行: {"type":"user","message":{...}} → 解析，返回 buffer: （空）

坑 4：os.fork() 跟 asyncio 不兼容

问题： 很多 PTY 教程用 os.fork() + os.exec() 来启动子进程。但在一个有 asyncio event loop 或多线程的父进程里 fork， 子进程会继承父进程的锁状态和线程表，导致死锁或状态不一致。我们的框架是给 FastAPI/asyncio 后端用的，fork 直接排除。
解决： 用 pty.openpty() 先创建 PTY 对，然后用 subprocess.Popen（内部用 posix_spawn 或安全的 fork+exec） 启动 CC，把 slave_fd 作为 stdin/stdout/stderr 传进去。父进程拿着 master_fd 做读写。

subprocess.Popen + start_new_session=True + close_fds=True 确保子进程在自己的 session 组里，文件描述符干净。

5. 多轮对话和会话池

因为 CC 进程一直活着，多轮对话天然连续——你不需要任何额外参数，直接发第二条 prompt 就行：

# 第一轮
async for event in session.send_prompt("记住数字 42"):
    ...
# 第二轮（CC 已经知道 42）
async for event in session.send_prompt("我刚才说的数字是什么？"):
    ...  # CC 会回答 "42"

当你需要同时运行多个 session（比如给 CCM 的多个 task 各开一个），SessionPool 帮你管理：

pool = SessionPool(config=PTYConfig(max_sessions=20))

# 每个 task 用自己的 session_id，pool 自动创建或复用
session_a = await pool.get_or_create("task-001", cwd="/project-a")
session_b = await pool.get_or_create("task-002", cwd="/project-b")

# 下次用同一个 session_id，直接复用已有进程
session_a_again = await pool.get_or_create("task-001", cwd="/project-a")
# session_a_again 就是 session_a，同一个 CC 进程
LRU 淘汰： 如果 session 数量达到上限（默认 20），pool 会自动关掉最久没用过的空闲 session，腾出位置给新的。 优先淘汰超过 5 分钟没用的 session；如果都在忙，淘汰最老的没在执行 prompt 的那个。

6. Channels：执行中途注入消息

正常的 prompt → response 流程够用的场景，用 JSONL 模式就行（上面讲的那些）。

但有时候你需要在 CC 正在执行工具的时候 给它发一条消息——比如"需求变了，别做了"或者"顺便也改一下这个文件"。 这时候 CC 已经在处理中，你不能再 send_prompt（它会等上一个 prompt 结束）。

Channels 是 CC 官方提供的 MCP（Model Context Protocol）扩展，允许在 CC 执行过程中往它的上下文注入消息。 CC 会在下一个工具调用的边界读到这条消息。

工作原理

你的后端 BridgeHub channel_server Claude Code │ (中心路由) (每个 session 一个, (正在工作中...) │ │ 作为 MCP 服务器运行) │ │ │ │ │ │ inject("需求变了") │ │ │ ├────────────────────────────────→│ │ │ │ │ HTTP POST /inject │ │ │ ├─────────────────────→│ │ │ │ │ JSON-RPC 通知 │ │ │ │ (写到 CC 的 stdin) │ │ │ ├─────────────────────→│ │ │ │ │ │ │ │ CC 在下一个工具 │ │ │ 调用边界读到消息: │ │ │ <channel>需求变了</channel> │ │ │ CC 调整后续行为

三个组件的角色

channel_server（MCP 服务器）
CC 启动时自动加载它（通过 .mcp.json 配置）。它是一个独立进程，通过 stdin/stdout 跟 CC 通信（JSON-RPC 协议）。 同时它开了一个 HTTP 端口接收外部消息。

当你发 inject 请求，它收到后通过 stdio 发一条 MCP notification 给 CC。CC 就看到注入的消息了。
BridgeHub（中心路由）
管理所有 session 的端口映射。你只需要 inject(session_id, "消息")，BridgeHub 知道这个 session 对应的 channel_server 在哪个端口，帮你转发。

怎么用

from claude_pty import SessionPool, BridgeHub

# 1. 启动 BridgeHub
bridge = BridgeHub(port=18000)
bridge.start()

# 2. 创建 session，开启 channels
pool = SessionPool(bridge=bridge)
session = await pool.get_or_create("task-1", "/project", channels=True)

# 3. 发 prompt（CC 开始执行，可能要花几分钟）
async for event in session.send_prompt("重构整个用户模块"):
    print(event)
    # 在 CC 执行期间，你可以随时注入消息：
    if need_to_change_requirement:
        await session.inject("需求更新：还需要支持 OAuth 登录")
不需要 Channels 的场景： CCM 现有的 agent 通信就是简单的优先级队列 + resume 模式——A agent 说完了，消息入队，B agent 下次执行时读取。 这种 "轮流说话" 的模式用普通的 send_prompt 就够了，不需要 Channels。

需要 Channels 的场景： CC 正在执行一个长任务（比如重构 20 个文件），执行到一半你想告诉它 "停下来，需求变了"。 或者多租户平台，需要给运行中的 CC 推送实时消息。

7. 权限中继：程序化控制工具执行

CC 执行工具（读文件、写文件、运行命令）前需要权限。在终端里这是交互式的弹窗，人类点 "Allow"。 在程序里，通常用 --dangerously-skip-permissions 全部放行。

但如果你想精细控制呢？比如 "允许读文件，但禁止写文件" 或 "禁止运行危险命令"。 这就是权限中继——通过 Channels MCP，CC 每次要用工具时先问你的程序，你的程序自动决定。

  1. 1
    CC 要用工具（比如 Edit 文件） CC 通过 stdio 发 permission_request 给 channel_server，包含工具名、描述、输入预览
  2. 2
    channel_server 转发给 BridgeHub HTTP POST，带上 session_id、request_id、tool_name
  3. 3
    BridgeHub 调用你注册的回调函数 你的代码根据规则判断 allow 或 deny，然后调用 resolve_permission()
  4. 4
    决定传回 CC BridgeHub → channel_server → CC。CC 收到 allow 就继续执行工具，收到 deny 就跳过。超时 120 秒默认 deny。 
# 示例：只允许读操作，禁止写操作
def handle_permission(session_id, request):
    tool = request["tool_name"]
    if tool in ("Read", "Bash", "Grep"):
        bridge.resolve_permission(session_id, request["request_id"], "allow")
    else:
        bridge.resolve_permission(session_id, request["request_id"], "deny")

bridge.on_permission_request(handle_permission)
大多数场景不需要用这个。 自动化场景（CCM 等）直接 --dangerously-skip-permissions 全部放行就行。 权限中继更适合多租户平台、沙箱环境、审计需求等场景。 它是框架提供的能力，但不强制使用。

8. 崩溃恢复

CC 进程可能因为各种原因崩溃（OOM、网络断开、内部错误）。框架的恢复流程：

  1. 1
    drain 线程检测到进程死亡 读 master_fd 时收到 EIO/EOF 错误，说明子进程已退出。设置 _child_dead 标志。
  2. 2
    如果正在执行 send_prompt，yield 一个 SESSION_CRASHED 事件 你的代码可以捕获这个事件做日志、告警等。
  3. 3
    下次调用 send_prompt 时，自动重启 指数退避等待（2 秒、4 秒、8 秒），然后用 claude --resume {session_id} 重新启动进程。 --resume 会恢复之前的对话上下文，CC 继续之前的工作。最多重试 3 次。
正常运行中... CC 进程: 突然崩溃（exit code 1） │ drain: os.read() → EIO 错误 → 设置 _child_dead │ session: 检测到 is_alive == False ├── yield PTYEvent(SESSION_CRASHED) │ ...你的代码继续运行，下次 send_prompt 时... │ session: _auto_resume() ├── 等 2 秒（指数退避） ├── 启动新进程: claude --resume {session_id} ├── CC 恢复之前的对话上下文 └── 继续工作

9. 代码示例

最简用法：发一条 prompt

from claude_pty import Session, PTYConfig

session = Session(cwd="/path/to/project")
await session.start()

async for event in session.send_prompt("帮我写一个 hello world"):
    print(event.event_type, event.content)

await session.stop()

多 session 并发（CCM 场景）

from claude_pty import SessionPool, PTYConfig

pool = SessionPool(config=PTYConfig(max_sessions=10))

# 每个 task 自己的 session
async def run_task(task_id, prompt, cwd):
    session = await pool.get_or_create(task_id, cwd)
    async for event in session.send_prompt(prompt):
        await save_to_db(task_id, event.to_dict())

# 并发跑多个 task
await asyncio.gather(
    run_task("task-1", "实现登录", "/project"),
    run_task("task-2", "写测试", "/project"),
)

await pool.stop_all()

Channels + 权限控制

from claude_pty import SessionPool, BridgeHub

bridge = BridgeHub(port=18000)
bridge.start()

# CC 调用工具时，自动允许读操作，禁止写操作
bridge.on_permission_request(lambda sid, req:
    bridge.resolve_permission(sid, req["request_id"],
        "allow" if req["tool_name"] in ("Read", "Bash") else "deny")
)

# CC 主动发消息给我们时
bridge.on_reply(lambda sid, text: print(f"CC says: {text}"))

pool = SessionPool(bridge=bridge)
session = await pool.get_or_create("task-1", "/project", channels=True)

# 正常发 prompt
async for event in session.send_prompt("分析一下代码"):
    print(event)

# 在 CC 执行中途注入消息
await session.inject("补充：重点看 auth 模块")

CCM 集成适配器（CCM 侧代码）

# CCM 只需写一个适配器，现有的 _process_event 不用任何改动
from claude_pty import SessionPool

class PTYAdapter:
    def __init__(self, pool: SessionPool):
        self.pool = pool

    async def send_prompt(self, task_id, instance_id, session_id, prompt, cwd):
        session = await self.pool.get_or_create(session_id, cwd=cwd)
        async for event in session.send_prompt(prompt):
            # event.to_dict() 输出格式 == StreamParser.parse_line() 输出格式
            await self._process_event(instance_id, task_id, event.to_dict())
