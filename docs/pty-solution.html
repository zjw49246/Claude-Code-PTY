1. 为什么替换 claude -p

CCM 原本每个回合都执行 claude -p "<prompt>" --output-format stream-json --resume <sid>。模式简单（进程退出 = 回合结束），但代价是：

问题说明
每轮冷启动每条 chat 消息、每个 loop 迭代都重新拉起整个 CC 进程
上下文重放--resume 每次从 JSONL 重建历史，会话越长越慢
无法中途干预只能 SIGINT 杀进程，不能注入消息、不能软中断
进程内状态丢失KV cache 等每轮归零
实测收益：端到端冒烟中，第二轮 chat 在同一进程上 7.8 秒完成（含模型响应时间），无任何冷启动开销。

2. 核心架构：PTY 只当宿主，消息走协议层

初版直觉「模拟人类打字 + 解析终端输出」被证伪：逐字符 stdin 会拆裂 UTF-8、换行提前提交、TUI 状态污染；ANSI 输出用光标定位渲染，不可能可靠解析。正确架构是输入输出全走结构化协议：

CCM instance_managerlaunch() 按 use_pty_mode 分流
flag 关 → 原 -p 路径零改动
▼ flag 开（claude provider）
CCMBackend 适配器process proxy 模拟 subprocess 接口
事件回灌 _process_event
▼
SessionPoolLRU ≤20 · 热复用 · drain_idle
▼ 每任务一个 Session（常驻 CC 进程）
输入：Channel 注入BridgeHub → channel_server(MCP)
→ notification 唤起新 turn
fallback: bracketed-paste stdin
宿主：PTY保活 · 启动应答
Esc 中断 · 活动信号
输出：JSONL 轮询~/.claude/projects/<hash>/<sid>.jsonl
normalize → StreamParser 对齐事件
▼ 回合结束 = system/turn_duration 哨兵 → proxy.complete(0) → dispatcher 无感知

3. 关键机制

3.1 启动对话框：双保险

主：spawn 前预写 ~/.claude.json 
"projects": { "<cwd>": {
  "hasTrustDialogAccepted": true,
  "hasClaudeMdExternalIncludesApproved": true,
  "enabledMcpjsonServers": ["pty-bridge"]
}}

原子写（tmp+rename），trust 框从根上不出现。

兜底：drain loop 通用自动应答

CC 用光标定位排版，可见空格非真空格 → 剥 ANSI + 折叠全部空白后匹配统一尾缀 Entertoconfirm，命中写 \r。600ms 冷却后主动 re-check（防连续对话框漏检），20s 窗口防运行期误判。

教训：枚举具体文案漏掉了 dev-channels 警告框，曾导致 channel 路径整体失效。

3.2 输入：Channel 注入（主）+ stdin（fallback）

send_prompt(text)
  ① bridge.inject(sid, text)   # HTTP → channel_server → MCP notification（重试 5×1s）
  ② 全部失败 → PTY stdin        # \x1b[200~ + text + \x1b[201~ + \r（bracketed paste）
承重墙（spike 验证）：CC idle 时收到 channel notification 会自动开启新 turn。注入消息以 <channel source="pty-bridge"> 进入上下文，完全绕开 TUI 输入层——prompt 永不触碰快捷键/斜杠补全/粘贴处理。

3.3 输出：JSONL 轮询 + 事件归一化

路径规则（实测）：~/.claude/projects/<re.sub(r'[^A-Za-z0-9]','-',cwd)>/<sid>.jsonl，所有非字母数字 → -。300ms 增量轮询，行缓冲处理 partial write，normalize() 输出与 CCM StreamParser 一致的事件（message / thinking / tool_use / tool_result + usage）。

3.4 回合结束：turn_duration 哨兵

{"type":"system","subtype":"turn_duration","durationMs":3348}   ← 每 turn 恰一条，在所有消息之后
为什么不用 stop_reason=end_turn：同一 turn 的 thinking 块和 text 块各占一行 JSONL、都带 end_turn，按它判断会提前截断丢失尾部消息。注意交互模式没有 result 事件（那是 -p 专属）。thinking 明文需账号 settings 开 showThinkingSummaries。

3.5 会话生命周期

场景行为
同 session_id 再次 launch池中活会话热复用：直接注入新 prompt
会话不在池中、磁盘有记录冷恢复：--resume <sid> 拉起（用 --session-id 会冲突）
进程崩溃指数退避 --resume 自愈（≤3 次）
号池换号migrate_session：JSONL 硬链接到新 CLAUDE_CONFIG_DIR + --resume
池满LRU 驱逐 idle 会话；mid-turn 永不驱逐
关闭 PTY 模式drain_idle：idle 全回收，mid-turn 跑完为止

4. 与 CCM 配套使用

4.1 一个 flag、一处分流

# instance_manager.launch() 开头
if provider == "claude" and self.pty_mode_enabled:
    return await self._launch_pty(...)   # → CCMBackend.launch_for_ccm
# 否则走原 claude -p 路径（零改动）
CCM 期望（-p 语义）PTY 模式实现
process.wait() / returncode_PTYProcessProxy：turn_duration → complete(0)
进程退出 = 回合结束消费完一个 turn 即「退出」，底层进程保活给下轮
stream-json → _process_eventPTYEvent.to_dict() 与 StreamParser 结构一致，DB/WebSocket 复用
SIGINT 停止stop() 分流：Esc 软中断 + 会话回收
dispatcher 超时 killproxy.kill() 真正回收会话并返回 -9
--mcp-config（$ skills）/ CLAUDE_CONFIG_DIR（号池）透传 PTYConfig

4.2 任务类型映射

场景PTY 模式
主任务执行新建常驻会话 + channel 注入
chat 多轮同一活会话直接注入——收益最大
loop / goal 轮次同一会话注入下一轮
goal 评估器 / monitor 子 agent保留 -p（一次性评估/独立机制，本就该用 -p）
codex provider不受影响

4.3 前端开关

导航栏右侧 PTY toggle → PUT /api/settings/runtime，无需重启后端。

  • 粒度 = 每次 launch（每条消息/每轮迭代都是一次 launch），切换立即影响下一次启动
  • 运行中的 turn 永不被切换，跑完为止，事件照常入库
  • 关闭时自动回收 idle PTY 进程（drain_idle）
  • 状态持久化 GlobalSettings 表，重启恢复；env USE_PTY_MODE 仅为初始缺省
  • 上下文无缝：两种模式共享 session_id/JSONL，切换后 --resume 接上历史
